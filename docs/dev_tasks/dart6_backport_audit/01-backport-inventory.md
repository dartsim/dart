# DART 6.20 backport inventory (main -> release-6.20)

> Sub-document of the `dart6_backport_audit` task. The task `README.md` is the
> single source of truth for scope, framing, and decisions; this file owns the
> **detailed evidence tables** only. Link to the README; do not duplicate it.
>
> Every row is an evidence-verified gap (or non-gap) between `origin/main`
> (DART 7) and `origin/release-6.20` (the unreleased DART 6.20 support lane).
> Each verdict is grounded in concrete `git`-level evidence (ancestry, content
> grep, file presence). Milestone for any resulting work: **DART 6.20**.

## How to read this inventory

- **Scope** distinguishes policy-mandated backports from maintainer-decision
  features. Per the release-roadmap backport policy, the **default** backport
  set is *compat-critical bug fixes + build fixes + ci-tooling that protect the
  6.x line*; everything tagged `feature-optional` is **outside** the auto-port
  scope and requires an explicit maintainer call (several are Gazebo/URDF-compat
  adjacent, so they are worth a decision — but they are **not** auto-backport).
  Note: the one build item here (#3108) is tagged `feature-optional`
  (build-convenience), not an in-scope build fix — see section 2.
- **Verdict** is the verification outcome: `gap-confirmed` (present on main,
  absent on release-6.20 by content) or `not-applicable` (no main->release gap
  to port).
- **Recommendation** is the suggested disposition, not a commitment.
- **Path nuance:** `origin/main` is DART 7 snake_case (`dart/.../foo_bar.cpp`);
  `origin/release-6.20` is DART 6 PascalCase (`dart/.../FooBar.cpp`). Most ports
  must be re-expressed against the PascalCase layout. Where the original commit
  predates the snake_case migration, the merged-main file may have been
  *renamed* after the fact — the row notes call this out.
- **No performance rows here by design.** There is **no actionable perf port
  gap**: release-6.20 met its perf needs independently (#3028 / #3023 / #3033 /
  #3037 / #3040 / #3125 plus the `issue_3056_dart6_performance` task). Main's
  DART-7-only perf work (#2504 / #2537 / #3011 / #3096 and the
  `dart/collision/native` stack) lives on paths that **do not exist** on
  release-6.20 and must **not** be ported. See the task `README.md`.

## Validation gate (per surface, before any port lands)

For each ported surface: build + the focused unit test for the touched module,
plus the gz-physics gate (`pixi run -e gazebo test-gz`) for any
collision / constraint / parser / default-solver surface, and always
`pixi run lint`.

## Appendix B — The bulk of main is DART-7-only and out of scope

The vast majority of `origin/main`'s commits since the 6.x branch point are
**DART-7-only architecture work** that does not map onto release-6.20 at all and
is therefore **out of scope** for this inventory: the ECS World rewrite (the
classic `dart/simulation/World.{hpp,cpp}` no longer exists on main, replaced by
`world.hpp`/`world_options.hpp`), the native collision rewrite
(`dart/collision/native`, owned as a separate big-ticket lever — see below), the
nanobind dartpy migration, the Filament renderer, the C++23 baseline, the
sweeping snake_case file/symbol renames, and the deprecated-API removals. These
touch paths that do not exist on release-6.20 and carry DART 7 semantics; they
must **not** be ported. The single largest DART-7->6.20 lever, the
**native-collision port** (porting `dart/collision/native` so FCL/Bullet/ODE
become optional and fcl drops from core), is **owned by the
`dart6_dependency_minimization` task (lane 3) and `issue_3056`**, not by this
audit — only its first slice (#3123) has merged. This inventory **references**
that lever and does not re-plan it. Likewise, release-6.20's performance needs
were met **independently** (#3028 / #3023 / #3033 / #3037 / #3040 / #3125 + the
`issue_3056_dart6_performance` task), so there is **no perf port gap** and main's
DART-7-only perf (#2504 / #2537 / #3011 / #3096 + the native-collision perf
stack) must **not** be chased onto release-6.20.
